# IM-60 운용자 매뉴얼

## 1. 문서 개요

본 문서는 AgiTeamBuilder Desktop 운영자와 개발자를 위한 앱 빌드, 인스톨러 생성, 환경 설정, 로그인, 진단 및 문제 해결 절차를 정의한다. 대상 앱은 Tauri v2 + Rust + Vue 3 + TypeScript 기반 데스크톱 애플리케이션이다.

## 2. 운영자 역할

| 역할 | 책임 |
|---|---|
| 개발 운영자 | 로컬 개발 환경 구성, 앱 빌드, 진단 도구 실행 |
| 릴리즈 운영자 | 플랫폼별 인스톨러 생성, 서명, 배포 산출물 보관 |
| 프로젝트 운영자 | workspace 설정, `agiteam.json` 가져오기, credential 등록 |
| 보안 운영자 | API 키 평문 저장 여부 점검, OS Credential Vault 사용 확인 |

## 3. 사전 준비

DV-10 환경구성서 기준으로 다음 도구가 설치되어 있어야 한다.

| 도구 | 기준 |
|---|---|
| Rust | rustup 최신 stable toolchain |
| Cargo | Rust stable toolchain 포함 |
| Tauri CLI | v2 계열 |
| Node.js | 20 LTS 이상 |
| pnpm | 최신 stable 권장 |
| macOS | Xcode Command Line Tools |
| Windows | Visual Studio C++ Build Tools, Windows SDK, WebView2 Runtime |

설치 확인:

```bash
rustc --version
cargo --version
cargo tauri --version
node --version
pnpm --version
```

## 4. 소스 준비 및 의존성 설치

프로젝트 저장소를 받은 뒤 의존성을 설치한다.

```bash
cd <agiteambuilder-desktop>
pnpm install
```

Rust 의존성은 Tauri 빌드 시 `src-tauri/Cargo.toml` 기준으로 자동 해석된다. 별도 확인이 필요한 경우 다음 명령을 사용한다.

```bash
cd src-tauri
cargo check
```

## 5. 앱 실행 및 빌드 명령

### 5.1 개발 모드 실행

```bash
cargo tauri dev
```

확인 항목:

| 항목 | 기대 결과 |
|---|---|
| 앱 창 | AgiTeamBuilder Desktop UI 표시 |
| Vue 개발 서버 | Tauri dev 프로세스와 함께 정상 기동 |
| Rust backend | Tauri command 호출 가능 |
| 로그 | secret 원문 노출 없음 |

### 5.2 운영 빌드

```bash
cargo tauri build
```

빌드 성공 후 산출물은 `src-tauri/target/release/` 및 Tauri bundle 출력 디렉터리에서 확인한다.

## 6. 플랫폼별 인스톨러 생성

### 6.1 Windows 인스톨러

Windows 인스톨러는 Windows 빌드 머신에서 생성한다.

```powershell
cargo tauri build --target x86_64-pc-windows-msvc
```

필수 조건:

| 항목 | 기준 |
|---|---|
| Runner 또는 OS | Windows |
| Toolchain | MSVC |
| Windows SDK | 설치 필요 |
| WebView2 Runtime | 설치 또는 대상 환경 포함 필요 |
| 코드 서명 | 배포 빌드 시 인증서 적용 |

운영 확인:

1. 인스톨러 파일 생성 여부를 확인한다.
2. 새 Windows 테스트 환경에 설치한다.
3. 앱 실행, 프로젝트 열기, credential 저장, 팀 부팅 흐름을 확인한다.
4. 제거 후 사용자 데이터 보존 정책을 확인한다.

### 6.2 macOS DMG

macOS DMG는 macOS 빌드 머신에서 생성한다.

```bash
cargo tauri build
```

필수 조건:

| 항목 | 기준 |
|---|---|
| Runner 또는 OS | macOS |
| Xcode Command Line Tools | 설치 필요 |
| Apple Developer 인증서 | 배포 빌드 시 필요 |
| Notarization | 외부 배포 시 필요 |

운영 확인:

1. `.app` 및 `.dmg` 산출물 생성 여부를 확인한다.
2. Gatekeeper 경고 여부를 확인한다.
3. 서명과 notarization 결과를 확인한다.
4. 새 macOS 테스트 환경에 설치 후 앱 실행을 확인한다.

## 7. CI/CD 운영

GitHub Actions는 OS별 native runner를 사용한다.

| 대상 | Runner | 빌드 명령 |
|---|---|---|
| Windows | `windows-latest` | `cargo tauri build --target x86_64-pc-windows-msvc` |
| macOS | `macos-latest` | `cargo tauri build` |

기본 단계:

1. checkout
2. Node.js 20 설정
3. pnpm 설정
4. Rust stable toolchain 설정
5. `pnpm install --frozen-lockfile`
6. `cargo tauri build`
7. 산출물 업로드
8. 배포 빌드인 경우 코드 서명 및 notarization 수행

CI/CD secret:

| Secret | 용도 |
|---|---|
| `WINDOWS_CERTIFICATE_BASE64` | Windows 코드 서명 인증서 |
| `WINDOWS_CERTIFICATE_PASSWORD` | Windows 인증서 비밀번호 |
| `APPLE_CERTIFICATE_BASE64` | macOS Developer ID 인증서 |
| `APPLE_CERTIFICATE_PASSWORD` | macOS 인증서 비밀번호 |
| `APPLE_TEAM_ID` | Apple Developer Team ID |
| `APPLE_API_KEY` | App Store Connect API Key |
| `APPLE_API_KEY_ID` | App Store Connect Key ID |
| `APPLE_API_ISSUER` | App Store Connect Issuer ID |

비밀값은 저장소에 커밋하지 않는다.

## 8. 환경 설정

### 8.1 프로젝트 열기

1. 앱을 실행한다.
2. 런처 화면에서 “기존 프로젝트 열기”를 선택한다.
3. 프로젝트 루트의 `agiteam.json`을 선택한다.
4. 앱이 설정 파일을 검증한다.
5. 오류가 없으면 프로젝트 설정 화면으로 이동한다.

### 8.2 프로젝트 설정 항목

| 항목 | 설명 |
|---|---|
| 프로젝트 이름 | 내부 식별자 |
| 표시 이름 | 앱 헤더에 표시되는 프로젝트명 |
| workspace root | `agiteam.json`, `project_state.yaml`, `brain/`의 기준 경로 |
| persona directory | 역할별 persona 디렉터리 경로 |
| common persona | 공통 persona 파일 경로 |
| team roles | PM 및 팀원 역할, agent, command, layout |
| timeout | readyTimeout, postLaunchDelay, readySignalTimeout 등 |

### 8.3 상태 파일 설정

앱은 workspace root에서 `project_state.yaml`을 로드한다.

| 필드 | 용도 |
|---|---|
| `business_type` | 업무 유형 |
| `current_mode` | 현재 협업 모드 |
| `milestone` | 현재 마일스톤 |
| `wbs_track` | WBS 트랙 |
| `milestones` | 마일스톤 목록 |

YAML 파싱 오류가 있으면 앱은 저장을 차단하고 오류를 표시해야 한다.

## 9. 로그인 및 자격증명 관리

### 9.1 기본 원칙

- API 키와 토큰은 평문 파일에 저장하지 않는다.
- macOS는 Keychain, Windows는 Windows Credential Manager를 사용한다.
- `agiteam.json`에는 credential reference 또는 계정 별칭만 저장한다.
- UI와 로그에는 secret 원문을 표시하지 않는다.

### 9.2 에이전트 로그인 확인

| 대상 | 확인 방식 |
|---|---|
| Claude Code | 기존 로그인 상태 또는 OS credential 확인 |
| Codex | Codex 로그인 상태 확인 |
| Gemini | Gemini CLI 또는 provider credential 확인 |
| 레드마인 | 역할별 API 키를 OS Credential Vault에 저장 후 연결 테스트 |

### 9.3 레드마인 설정

기본 레드마인 URL:

```text
http://211.117.60.5:8080/
```

운영 주의:

1. 레드마인은 내부망 HTTP 예외로 취급한다.
2. 외부 provider API에는 HTTP를 허용하지 않는다.
3. 역할별 API 키는 OS Credential Vault에 저장한다.
4. 연결 테스트 실패 시 URL, 네트워크, API 키 권한을 순서대로 확인한다.

## 10. 진단 도구 doctor 활용

### 10.1 실행 목적

doctor는 운영자가 앱 실행 전후 환경 문제를 빠르게 확인하기 위한 진단 도구다.

점검 범위:

| 항목 | 점검 내용 |
|---|---|
| 개발 도구 | Rust, Cargo, Tauri CLI, Node.js, pnpm 설치 여부 |
| OS 의존성 | Xcode Command Line Tools, Visual Studio Build Tools, WebView2 |
| workspace | `agiteam.json`, `project_state.yaml`, `brain/` 존재 여부 |
| 설정 | JSON/YAML schema validation |
| credential | OS Credential Vault 접근 및 필수 credential 존재 여부 |
| 네트워크 | provider API HTTPS 도달성, 레드마인 내부망 HTTP 예외 도달성 |

### 10.2 실행 방법

구현 방식이 Tauri command로 제공되는 경우 앱의 진단 화면에서 doctor를 실행한다.

```text
Settings > Diagnostics > Run Doctor
```

CLI 보조 명령이 제공되는 경우 다음 형식을 사용한다.

```bash
cargo tauri dev
```

앱 실행 후 진단 화면에서 결과를 확인한다. 별도 standalone doctor binary가 제공되면 릴리즈 노트에 명령명을 기록한다.

### 10.3 결과 해석

| 상태 | 의미 | 조치 |
|---|---|---|
| PASS | 정상 | 추가 조치 없음 |
| WARN | 실행 가능하나 운영 위험 있음 | 권장 조치 확인 |
| FAIL | 필수 조건 불충족 | 해당 항목 해결 전 운영 중단 |

## 11. 문제 해결 가이드

### 11.1 `cargo tauri dev` 실패

점검 순서:

1. `cargo tauri --version` 확인
2. `pnpm install` 재실행
3. `cd src-tauri && cargo check` 실행
4. Node.js 버전이 20 LTS 이상인지 확인
5. macOS는 Xcode Command Line Tools, Windows는 Visual Studio Build Tools 설치 확인

### 11.2 Windows 빌드 실패

원인 후보:

| 증상 | 조치 |
|---|---|
| `link.exe` 또는 `cl.exe` 없음 | Visual Studio C++ Build Tools 설치 |
| Windows SDK 없음 | Windows 10/11 SDK 설치 |
| WebView2 관련 오류 | WebView2 Runtime 설치 확인 |
| target 오류 | `rustup target add x86_64-pc-windows-msvc` 실행 |

### 11.3 macOS 빌드 또는 실행 실패

원인 후보:

| 증상 | 조치 |
|---|---|
| compiler not found | `xcode-select --install` 실행 |
| Gatekeeper 경고 | 코드 서명 및 notarization 상태 확인 |
| 권한 오류 | 앱 권한과 workspace 경로 접근 권한 확인 |
| Apple Silicon/Intel 이슈 | 대상 architecture와 runner 확인 |

### 11.4 프로젝트 설정 로드 실패

점검 순서:

1. `agiteam.json`이 유효한 JSON인지 확인한다.
2. 필수 필드가 누락되지 않았는지 확인한다.
3. `brain/` 경로가 workspace root 하위인지 확인한다.
4. startupFiles 경로가 존재하는지 확인한다.
5. API 키 원문이 포함되어 있으면 OS Credential Vault로 분리한다.

### 11.5 `project_state.yaml` 파싱 실패

점검 순서:

1. YAML 들여쓰기 오류를 확인한다.
2. `business_type`, `current_mode`, `milestone`, `wbs_track` 필드를 확인한다.
3. milestones 리스트 형식을 확인한다.
4. 앱 저장 전 백업본으로 복구한다.

### 11.6 credential 저장 실패

점검 순서:

| OS | 조치 |
|---|---|
| macOS | Keychain 접근 권한, 잠금 상태, 앱 서명 상태 확인 |
| Windows | Windows Credential Manager 접근 권한, 사용자 계정 정책 확인 |

평문 파일 fallback 저장은 금지한다.

### 11.7 네트워크 연결 실패

점검 순서:

1. provider API URL이 HTTPS인지 확인한다.
2. 프록시 또는 방화벽 정책을 확인한다.
3. 레드마인 URL `http://211.117.60.5:8080/` 접근 가능 여부를 확인한다.
4. 레드마인 API 키 권한과 프로젝트 접근 권한을 확인한다.
