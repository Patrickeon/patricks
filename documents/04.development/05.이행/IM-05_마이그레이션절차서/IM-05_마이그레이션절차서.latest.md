# IM-05 마이그레이션 절차서

## 1. 문서 개요

본 문서는 기존 `agiteam.sh` 기반 AgiTeamBuilder 운영 환경을 AgiTeamBuilder Desktop으로 전환하는 절차를 정의한다. 전환 대상은 `agiteam.json`, `brain/` 페르소나 디렉터리, `project_state.yaml`, cmux 기반 작업 흐름이다.

## 2. 전환 범위

| 구분 | AS-IS | TO-BE |
|---|---|---|
| 실행 방식 | `agiteam.sh` Bash CLI | Tauri v2 Desktop 앱 |
| UI/작업 패널 | cmux workspace/pane | Vue 3 기반 앱 워크스페이스 |
| 설정 파일 | `agiteam.json` 직접 편집 | 앱 설정 화면에서 가져오기/편집 |
| 상태 파일 | `project_state.yaml` CLI 파싱 | Rust YAML parser 기반 로드/저장 |
| 페르소나 | `brain/` 디렉터리 직접 참조 | 앱 workspace 설정으로 경로 지정 |
| 자격증명 | shell 환경, 로컬 파일, 도구별 저장소 혼재 | OS Credential Vault |

## 3. 사전 요구사항

DV-10 환경구성서 기준 개발 환경을 준비한다.

| 항목 | 기준 |
|---|---|
| Rust | rustup 최신 stable toolchain |
| Tauri CLI | v2 계열, `cargo install tauri-cli --locked` |
| Node.js | 20 LTS 이상 |
| 패키지 매니저 | pnpm 권장 |
| macOS | Xcode Command Line Tools |
| Windows | Visual Studio C++ Build Tools, WebView2 Runtime |

설치 확인:

```bash
rustc --version
cargo --version
cargo tauri --version
node --version
pnpm --version
```

## 4. 마이그레이션 전 백업

전환 전 기존 프로젝트 루트에서 다음 파일과 디렉터리를 백업한다.

```bash
cd <existing-agiteam-project>
mkdir -p _migration_backup
cp agiteam.json _migration_backup/agiteam.json.$(date +%Y%m%d%H%M%S)
cp project_state.yaml _migration_backup/project_state.yaml.$(date +%Y%m%d%H%M%S)
cp -R brain _migration_backup/brain.$(date +%Y%m%d%H%M%S)
```

백업 대상:

| 대상 | 목적 |
|---|---|
| `agiteam.json` | 프로젝트, 역할, agent command, startupFiles 설정 보존 |
| `project_state.yaml` | 현재 사업 유형, 모드, 마일스톤, WBS 상태 보존 |
| `brain/` | 역할별 persona, 공통 persona, 프로젝트별 인스턴스 보존 |

## 5. 기존 `agiteam.json` 가져오기

### 5.1 가져오기 절차

1. AgiTeamBuilder Desktop을 실행한다.
2. 런처 화면에서 “기존 프로젝트 열기”를 선택한다.
3. 기존 프로젝트 루트의 `agiteam.json`을 선택한다.
4. 앱이 JSON schema validation을 수행한다.
5. 검증 결과를 확인한 뒤 앱 설정으로 가져온다.

### 5.2 가져오기 검증 항목

| 항목 | 검증 기준 |
|---|---|
| 프로젝트 이름 | `project.name`, `project.displayName` 값 존재 |
| workspace 설정 | workspace name/color 값 존재 |
| PM 설정 | PM role, agent, command, startupFiles 존재 |
| 팀원 설정 | team 배열의 role/name/agent/command/layout 값 유효 |
| persona 설정 | persona dir/commonFile 경로 유효 |
| timeout 설정 | readyTimeout, postLaunchDelay 등 숫자형 값 유효 |

### 5.3 API 키 이관 정책

`agiteam.json`에는 API 키 원문을 저장하지 않는다. 가져오기 중 다음 필드명 또는 값 패턴이 발견되면 앱 설정 저장을 차단하고 OS Credential Vault 저장 절차로 전환한다.

| 탐지 대상 | 처리 |
|---|---|
| `apiKey`, `api_key` | 원문 저장 차단, credential 입력 화면으로 이동 |
| `token`, `accessToken` | 원문 저장 차단, credential 입력 화면으로 이동 |
| `secret`, `password` | 원문 저장 차단, credential 입력 화면으로 이동 |
| 긴 난수형 문자열 | 사용자가 secret 여부 확인 후 credential로 분리 |

저장 방식:

| OS | 저장소 |
|---|---|
| macOS | Keychain |
| Windows | Windows Credential Manager |

## 6. `brain/` 페르소나 디렉터리 경로 설정

### 6.1 기본 경로

기존 프로젝트 구조를 유지하는 경우 앱 workspace root를 기존 프로젝트 루트로 지정하고, `brain/`은 상대 경로로 참조한다.

```text
<workspace-root>/
├── agiteam.json
├── project_state.yaml
└── brain/
    ├── PM/
    ├── DevOps/
    ├── Architect/
    ├── DeveloperBE/
    ├── DeveloperFE/
    ├── Designer/
    └── QA/
```

### 6.2 설정 절차

1. 프로젝트 설정 화면에서 workspace root를 확인한다.
2. persona directory 값을 `brain` 또는 기존 커스텀 경로로 설정한다.
3. common persona 파일 경로를 확인한다.
4. 역할별 `persona.md` 존재 여부를 앱 진단 도구로 확인한다.
5. 누락 파일이 있으면 마이그레이션을 중단하고 백업본에서 복구한다.

### 6.3 경로 검증 기준

| 항목 | 기준 |
|---|---|
| workspace root | 사용자가 선택한 프로젝트 루트 하위 경로만 허용 |
| persona directory | canonicalize 후 workspace root prefix 일치 |
| startupFiles | workspace root 외부 파일 참조 금지 |
| role persona | `brain/<역할>/persona.md` 파일 존재 |

## 7. `project_state.yaml` 이관

### 7.1 가져오기 절차

1. 앱이 workspace root에서 `project_state.yaml`을 탐색한다.
2. Rust YAML parser로 파일을 로드한다.
3. 필수 스칼라와 milestones 리스트를 검증한다.
4. 유효한 값은 앱 상태 바와 프로젝트 상태 편집 화면에 반영한다.
5. YAML 파싱 오류가 있으면 저장을 차단하고 오류 위치를 표시한다.

### 7.2 필수 확인 항목

| 항목 | 설명 |
|---|---|
| `business_type` | 프로젝트 업무 유형 |
| `current_mode` | 현재 협업 모드 |
| `milestone` | 현재 마일스톤 |
| `wbs_track` | WBS 트랙 |
| `milestones` | 마일스톤 목록 |

### 7.3 이관 후 저장 정책

- 앱에서 수정한 `project_state.yaml`은 YAML parser를 통해 구조화 저장한다.
- grep/sed 기반 부분 수정은 사용하지 않는다.
- 저장 전 백업 파일을 생성한다.
- 파싱 실패 상태에서는 기존 파일을 덮어쓰지 않는다.

## 8. cmux 의존성 제거 절차

### 8.1 제거 원칙

AgiTeamBuilder Desktop은 cmux workspace와 pane을 사용하지 않는다. 기존 cmux 명령은 앱 내부 Rust 서비스와 Vue UI로 대체한다.

| AS-IS cmux 명령 | TO-BE 대체 기능 |
|---|---|
| `cmux send` | 팀원 패널 입력창 + Rust `send_agent_message` |
| `cmux send-key Enter` | 전송 버튼 또는 단축키 |
| `cmux read-screen` | 앱 이벤트 스트림 및 패널 메시지 렌더링 |
| `cmux identify` | 앱 내부 workspace/session 상태 관리 |
| `cmux list-pane-surfaces` | 앱 팀 구성 패널 목록 |
| `cmux rename-workspace` | 앱 워크스페이스 헤더 |
| `cmux workspace-action set-color` | 앱 workspace 테마 표시 |
| `cmux browser` | 앱 내장 브라우저 제어 기능 |

### 8.2 운영 절차

1. 기존 `agiteam.sh` 자동 실행 스크립트, shell alias, launcher에서 cmux 호출을 제거한다.
2. PM과 팀원 운영 절차 문서에서 surface 번호 기반 지시를 앱 패널 기반 지시로 교체한다.
3. 브라우저 제어 기능은 앱 내장 browser command 또는 별도 진단 도구로 대체한다.
4. 전환 기간에는 기존 `agiteam.sh`를 백업 전용으로 보존하되 신규 작업에는 사용하지 않는다.
5. 앱 기반 부팅과 메시지 송수신 검증이 완료되면 cmux 설치 여부를 필수 조건에서 제거한다.

### 8.3 제거 검증

```bash
rg -n "cmux|surface:" <workspace-root>
```

검증 기준:

| 위치 | 기준 |
|---|---|
| 운영 스크립트 | 신규 실행 경로에서 cmux 호출 없음 |
| 사용자 가이드 | surface 번호 지시 제거 또는 legacy 문맥으로만 표시 |
| 앱 설정 | cmux binary path 설정 없음 |
| 진단 도구 | cmux 설치 여부를 필수 검사로 취급하지 않음 |

## 9. 전환 후 검증 절차

### 9.1 앱 실행 검증

```bash
cargo tauri dev
```

확인 항목:

| 항목 | 기대 결과 |
|---|---|
| 프로젝트 열기 | 기존 `agiteam.json` 선택 가능 |
| 설정 검증 | schema validation 결과 표시 |
| persona 로드 | 역할별 persona 파일 탐색 성공 |
| 상태 표시 | `project_state.yaml` 값이 앱 상태 바에 표시 |
| credential | API 키 원문 없이 OS Credential Vault 참조 |

### 9.2 빌드 검증

```bash
cargo tauri build
```

빌드 산출물은 OS별 runner 또는 로컬 개발 환경에서 확인한다.

## 10. 롤백 절차

전환 중 치명 오류가 발생하면 다음 절차로 롤백한다.

1. AgiTeamBuilder Desktop을 종료한다.
2. `_migration_backup`의 `agiteam.json`, `project_state.yaml`, `brain/` 백업본을 원위치로 복구한다.
3. 기존 `agiteam.sh` 실행 권한과 shell alias를 복구한다.
4. API 키가 OS Credential Vault로 일부 이관된 경우 중복 저장 상태를 확인한다.
5. 롤백 사유와 오류 로그를 이슈로 등록한다.

## 11. 완료 기준

| 기준 | 완료 조건 |
|---|---|
| 설정 이관 | `agiteam.json`이 앱에서 로드되고 저장 가능 |
| 상태 이관 | `project_state.yaml`이 파싱되고 앱 UI에 표시 |
| persona 이관 | `brain/` 경로와 역할별 persona 확인 완료 |
| cmux 제거 | 신규 실행 경로에서 cmux 호출 없음 |
| 보안 | API 키 원문이 설정 파일에 저장되지 않음 |
| 빌드 | `cargo tauri build` 성공 |
