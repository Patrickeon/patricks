### 1.3 시험 원칙

1. 요구사항 ID(FR/NFR/PR)를 시험 항목의 기본 추적 단위로 사용한다.
2. 화면 시험은 DS-10의 Screen-01~07과 내장 브라우저 패널을 기준으로 한다.
3. 백엔드 시험은 DS-20의 Rust service, command, event, error code 단위를 기준으로 한다.
4. 모든 주요 기능은 정적 재현과 동적 재현을 분리한다.
5. macOS와 Windows 양쪽에서 동일 기능, 동일 상태 전이, 동일 오류 메시지 기준을 검증한다.
6. 자격증명, API key, token 원문은 시험 로그와 증거 파일에 남기지 않는다.

## 3. 시험 유형 및 방법

### 3.1 단위 테스트 - Rust 백엔드

| 대상 | 주요 검증 항목 | 예시 |
|------|----------------|------|
| ConfigService | `agiteam.json`, `project_state.yaml` parsing, schema validation, 오류 코드 | FR-0801 |
| PersonaBundleService | Shared persona + role persona 병합, 비-PM 부팅 대기 규칙 append, content hash | FR-0205, FR-0802 |
| AgentSessionService | lifecycle state 전이, message 저장, provider event 정규화 | FR-0804 |
| FileDocumentService | workspace root path 검증, `_archive` 백업, `.latest.md` 갱신, frontmatter parsing | FR-0504, FR-0807 |
| CredentialStoreService | OS credential vault 저장/조회/삭제, secret 원문 미반환 | FR-0705, NFR-0301 |
| HealthCheckService | CLI 존재, provider endpoint 도달성, workspace 구조 검증 | FR-0202, FR-0707 |
| Redmine client | issue CRUD, status transition, notes, done_ratio, 오류 정규화 | FR-0601~FR-0606 |
| Provider Adapter | Claude/OpenAI/Gemini request/streaming/tool event 변환 | FR-0803, FR-0809 |

실행 기준:

```bash
cargo test --workspace
```

합격 기준:

- 전체 Rust 단위 테스트 PASS.
- 오류 케이스는 DS-20 §5.3의 `AppError` 구조와 대표 오류 코드를 반환.
- secret 포함 가능 문자열은 모든 로그/오류/assertion snapshot에서 masking.

### 3.2 통합 테스트 - Tauri IPC

| 대상 | 주요 검증 항목 |
|------|----------------|
| Command 호출 | DS-20 §5.1 command 목록의 request/response schema, 성공/실패 경로 |
| Event stream | `agent:status_changed`, `agent:message_delta`, `document:changed`, `health:completed` 발생 순서 |
| Frontend wrapper | `src/ipc/*.ts` typed wrapper를 통한 invoke 호출, 직접 산발 호출 방지 |
| 상태 반영 | backend event가 Pinia store mutation으로 반영되는지 확인 |
| 보안 경계 | renderer가 credential secret 또는 workspace root 외부 파일에 접근하지 못하는지 확인 |

실행 기준:

```bash
cargo test --workspace --test '*ipc*'
npm run test
```

합격 기준:

- 각 IPC command는 정상 입력, 필수값 누락, 권한/경로 위반, session state 오류 케이스를 포함한다.
- 장시간 작업은 invoke 응답과 event stream이 분리되어 UI가 block되지 않는다.
- 모든 path 입력은 canonicalize 후 workspace root 하위로 제한된다.

### 3.4 정적 재현 + 동적 재현 2단계 방식

RACI매트릭스 §7.5의 판정 라벨 중 본 시험계획서의 기본 목표는 ③ 정적 재현과 ④ 동적 재현이다.

| 단계 | 정의 | 수행 방법 | 증거 |
|------|------|-----------|------|
| 정적 재현 | 코드·문서 인용만으로 실행 가능한 시나리오를 구성 | AN-40 요구사항 ID, DS-10 화면/라우트, DS-20 command/event/service를 시험시나리오 Given/When/Then에 연결 | TS-15 시험시나리오, 요구사항 추적표, 코드/문서 인용 |
| 동적 재현 | 재현 환경에서 실제 앱 또는 테스트 harness 실행 결과 관찰 | macOS/Windows에서 unit/integration/E2E 실행, 화면/로그/snapshot 수집 | 테스트 로그, Playwright trace, 스크린샷, cargo/npm test 결과 |

판정 기준:

- PASS: 정적 재현과 동적 재현이 모두 성공하고, 결과가 요구사항/설계 기준과 일치.
- CONDITIONAL PASS: 정적 재현은 가능하나 외부 서비스 mock 또는 제한 환경으로 동적 재현 범위가 제한됨. 제한 사유와 후속 재검증 조건을 기록.
- FAIL: 정적 재현 시나리오 자체가 불가능하거나, 동적 실행 결과가 요구사항/설계 기준과 불일치.
- BLOCKED: 선행 산출물, 테스트 환경, credential, 외부 서비스 접근 권한 부재로 실행 불가. PM에게 즉시 보고.

## 6. 결함 관리 기준

### 6.1 심각도

| 심각도 | 기준 | 예 |
|--------|------|----|
| Critical | 앱 실행/부팅/데이터 보존/credential 보호가 불가능 | 앱 시작 실패, API key 평문 노출, `.latest.md` 손상 |
| High | P1 핵심 흐름이 실패하거나 요구사항과 불일치 | 팀 부팅 실패, 메시지 전송 실패, workspace guard 우회 |
| Medium | 우회 가능한 기능 실패 또는 P2 기능 불일치 | Redmine 필터 오류, diff 표시 오류 |
| Low | 문구, 시각 피드백, 경미한 UX 문제 | 토스트 문구 누락, spacing 차이 |

### 6.2 결함 등록 필수 항목

| 항목 | 내용 |
|------|------|
| 제목 | `[결함] <요구사항ID> <현상 요약>` |
| 재현 절차 | Given/When/Then 형식, OS와 build hash 포함 |
| 기대 결과 | AN-40/DS-10/DS-20 기준 인용 |
| 실제 결과 | 로그, 스크린샷, trace, API response |
| 심각도 | Critical/High/Medium/Low |
| 원복 | 테스트 데이터 변경 시 SQL 또는 파일 원복 절차 |

### 6.3 레드마인 상태 처리

1. QA가 결함 발견 시 tracker_id=1로 결함 등록.
2. 담당자가 수정 완료 보고 시 status_id=3(해결)로 전환.
3. QA 재검증 PASS 후 PM에게 완료 검토 요청.
4. PM이 최종 status_id=5(완료) 처리.
