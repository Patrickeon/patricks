# TS-70 보안 취약성 점검 결과서

## 1. 문서 개요

본 문서는 AgiTeamBuilder Desktop의 Tauri v2 앱 전환 설계에 대한 보안 취약성 점검 결과를 기록한다. 점검 범위는 Tauri capabilities, OS Credential Vault, 네트워크 통신 정책, `agiteam.json` 비밀값 저장 금지, WebView 콘텐츠 보안 정책(CSP)이다.

현재 점검은 DS-20 아키텍처설계서, DV-10 환경구성서, AN-40 요구사항명세서 NFR-03을 입력으로 수행한 문서 기반 점검이다. 실제 소스코드, `tauri.conf.json`, capability 파일, CI/CD secret 설정은 구현 완료 후 별도 실증 점검이 필요하다.

| DV-10 환경구성서 | Tauri v2 개발 환경, `src-tauri/`, `tauri.conf.json`, `capabilities/`, 비밀값 git 커밋 금지 |
| AN-40 요구사항명세서 | NFR-03 보안 요구사항: API 키 안전 저장, Tauri 최소 권한, 로컬 저장 데이터 보호, HTTPS 강제 |

| SEC-04 | `agiteam.json` 비밀값 저장 금지 | `agiteam.json`에는 API 키 원문을 저장하지 않고 credential reference만 저장 |
| SEC-05 | WebView CSP | inline script/object 허용 최소화, 외부 연결 도메인 allowlist 제한 |

## 5. 상세 점검 결과

### 5.1 SEC-01 Tauri capabilities 최소 권한 원칙

점검 내용:

- Tauri v2 capability로 frontend window/webview가 호출 가능한 command와 plugin 권한을 제한해야 한다.
- 기본 전체 권한은 비활성화하고, 화면별로 필요한 권한만 선언해야 한다.
- workspace 파일 접근과 credential 접근은 Rust command 내부에서 검증해야 한다.

문서 기준 확인:

- DS-20은 “기본 capability를 최소화하고, workspace 파일 접근과 credential 접근을 Rust command 내부 검증으로 제한”한다고 명시한다.
- DS-20은 command allowlist를 `tauri.conf.json` capability와 Rust command registry 양쪽에서 제한하도록 정의한다.
- DV-10 표준 폴더 구조에 `src-tauri/capabilities/`와 `src-tauri/tauri.conf.json`이 포함되어 있다.

판정:

| 항목 | 결과 |
|---|---|
| 최소 권한 원칙 문서화 | 적합 |
| 기본 전체 권한 비활성화 확인 | 구현 파일 미존재로 실증 보류 |
| 화면별 capability 세분화 | 구현 단계 확인 필요 |

조치:

1. `src-tauri/capabilities/default.json` 또는 동등 capability 파일에서 wildcard 권한 사용을 금지한다.
2. `core:default`, filesystem, shell, http plugin 권한은 기능별 필요 범위만 허용한다.
3. capability 변경 시 TS-70을 재점검하고 변경 이력을 남긴다.

### 5.2 SEC-02 OS Credential Vault 사용 여부

점검 내용:

- API 키, OAuth token, 레드마인 역할별 API 키는 평문 파일에 저장하지 않는다.
- macOS는 Keychain, Windows는 Windows Credential Manager를 사용한다.
- renderer/UI에는 secret 원문을 반환하지 않는다.

문서 기준 확인:

- AN-40 NFR-0301은 레드마인 API 키와 에이전트 자격증명을 OS 키체인에 저장하고 평문 파일 저장을 금지한다.
- DS-20은 `CredentialStoreService`를 OS Credential Vault 기반 서비스로 정의한다.
- DS-20은 secret 원문을 UI에 반환하지 않고 `CredentialRef`와 masked account 정보만 반환한다고 명시한다.

판정:

| 항목 | 결과 |
|---|---|
| OS Credential Vault 사용 원칙 | 적합 |
| API 키 평문 저장 금지 | 적합 |
| UI secret 원문 반환 금지 | 적합 |

조치:

1. 구현 시 `save_credential`, `delete_credential`, `validate_credential` command는 secret 원문을 로그와 event에 남기지 않는다.
2. 로컬 설정 파일에는 provider, account, credential id 또는 alias만 저장한다.
3. credential 저장 실패 시 평문 fallback 저장을 금지한다.

### 5.3 SEC-03 네트워크 요청 HTTPS 강제 여부

점검 내용:

- 외부 API 통신은 HTTPS만 허용한다.
- 레드마인 서버는 내부망 HTTP URL `http://211.117.60.5:8080/`를 예외로 기록한다.
- 예외 URL은 사용자 입력 임의 HTTP 허용으로 확장하지 않는다.

문서 기준 확인:

- AN-40 NFR-0304는 레드마인 제외 모든 외부 API 통신 HTTPS 강제를 요구한다.
- Shared 공용 레퍼런스는 레드마인 공용 접속 정보를 내부망 HTTP로 정의한다.
- DS-20은 Provider HTTP 호출을 Rust backend만 수행하도록 설계한다.

판정:

| 항목 | 결과 |
|---|---|
| 외부 API HTTPS 강제 | 적합 |
| 레드마인 HTTP 예외 기록 | 예외 있음 |
| renderer 직접 HTTP 호출 금지 | 적합 |

레드마인 HTTP 예외:

| 항목 | 값 |
|---|---|
| URL | `http://211.117.60.5:8080/` |
| 사유 | 내부망 레드마인 서버가 HTTP 프로토콜로 운영됨 |
| 제한 | 해당 host:port만 예외 허용, 외부 provider API에는 적용 금지 |
| 보완 | 운영 전 HTTPS 전환 또는 내부망 접근 통제 확인 권장 |

조치:

1. Rust HTTP client 생성 시 provider API는 `https://` scheme만 허용한다.
2. 레드마인 connector는 allowlist에 등록된 내부망 URL만 HTTP 예외로 허용한다.
3. 설정 화면에서 임의 HTTP URL을 입력하면 경고와 명시적 예외 등록 절차를 요구한다.

### 5.4 SEC-04 `agiteam.json` API 키 직접 저장 여부

점검 내용:

- 기존 `agiteam.json`은 프로젝트 설정과 팀 구성을 저장하되 API 키 원문을 저장하지 않는다.
- GUI 편집기는 credential reference 방식만 허용한다.
- import 과정에서 기존 파일에 API 키로 보이는 값이 있으면 가져오기를 차단하거나 OS Credential Vault 이관을 요구한다.

문서 기준 확인:

- AN-40 NFR-0303은 `agiteam.json`에 API 키 직접 저장 금지와 키체인 참조 방식을 요구한다.
- AN-40 FR-0705는 레드마인 역할별 API 키 입력값을 마스킹하고 OS 키체인에 저장한다고 정의한다.
- DS-20은 `ConfigLoader`가 `agiteam.json`을 schema validation으로 로드하도록 설계한다.

판정:

| 항목 | 결과 |
|---|---|
| API 키 직접 저장 금지 원칙 | 적합 |
| credential reference 방식 | 적합 |
| import 시 secret 탐지 | 구현 단계 확인 필요 |

조치:

1. `agiteam.json` schema에서 `apiKey`, `token`, `secret`, `password` 성격의 원문 필드를 금지한다.
2. import 시 API key pattern 탐지 결과를 사용자에게 표시하고 OS Credential Vault 저장으로 전환한다.
3. 저장 전 redaction 검사로 secret 원문 저장을 차단한다.

### 5.5 SEC-05 WebView 콘텐츠 보안 정책(CSP)

점검 내용:

- WebView는 로컬 Vite/Tauri bundle 콘텐츠만 렌더링하고, 외부 script 로드를 제한한다.
- `script-src`, `connect-src`, `img-src`, `style-src` 정책을 기능 단위로 최소화한다.
- provider API 호출은 Rust backend에서 수행하므로 renderer의 외부 `connect-src`는 최소화한다.

문서 기준 확인:

- DS-20은 provider HTTP 호출을 Rust backend만 수행한다고 정의한다.
- DS-20은 external URL 허용 도메인 정책을 별도 정의하고 기본은 시스템 브라우저 열기로 명시한다.
- DV-10은 `tauri.conf.json`을 Tauri 앱 설정, 번들링, 보안 정책 파일로 정의한다.

판정:

| 항목 | 결과 |
|---|---|
| CSP 정책 필요성 식별 | 적합 |
| 구체 CSP 값 확정 | 보완 필요 |
| 외부 provider 직접 연결 제한 | 적합 |

권장 CSP 초안:

```text
default-src 'self';
script-src 'self';
style-src 'self' 'unsafe-inline';
img-src 'self' data: asset:;
font-src 'self' data:;
connect-src 'self' ipc: http://ipc.localhost;
object-src 'none';
base-uri 'self';
frame-ancestors 'none';
```

조치:

1. `tauri.conf.json`의 security CSP 값을 구현 단계에서 확정한다.
2. 외부 웹 콘텐츠 표시가 필요한 경우 allowlist와 sandbox 정책을 별도 정의한다.
3. `unsafe-eval`, wildcard `*`, 임의 `http:` 허용은 금지한다.

## 7. 최종 판정

문서 기준 보안 설계는 AN-40 NFR-03 요구사항을 충족하는 방향으로 정의되어 있다. 단, 구현 파일이 아직 점검 대상에 포함되지 않았으므로 다음 항목은 구현 완료 후 필수 재점검한다.

1. `src-tauri/capabilities/` 권한 선언 파일
2. `src-tauri/tauri.conf.json` CSP 및 security 설정
3. `agiteam.json` schema validation과 secret 저장 차단 로직
4. CredentialStoreService의 OS별 저장/조회/삭제 동작
5. HTTP client의 HTTPS 강제 및 레드마인 HTTP allowlist 처리
