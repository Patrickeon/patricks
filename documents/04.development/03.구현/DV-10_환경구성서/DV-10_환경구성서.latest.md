# DV-10 환경구성서

## 1. 문서 개요

본 문서는 Tauri v2 + Rust + Vue 3 + TypeScript 기반 데스크톱 애플리케이션의 개발 환경 구성 기준을 정의한다. 로컬 개발 환경, 프로젝트 초기화, 빌드 명령, CI/CD 빌드 머신 구성, 서명 및 배포 준비 사항을 포함한다.

## 2. 확정 기술 스택

| 구분 | 기술 |
|---|---|
| 데스크톱 앱 프레임워크 | Tauri v2 |
| 백엔드 런타임 | Rust |
| 프론트엔드 프레임워크 | Vue 3 |
| 프론트엔드 언어 | TypeScript |
| 번들러 | Vite |
| 패키지 매니저 | pnpm 권장 |
| CI/CD 기준 | GitHub Actions |

## 3. 개발 환경 요구사항

### 3.1 공통 필수 도구

| 도구 | 권장 버전 또는 기준 | 비고 |
|---|---|---|
| Rust | rustup 최신 stable toolchain | `cargo`, `rustc` 포함 |
| Cargo | Rust stable toolchain 포함 버전 | Rust crate 빌드 및 Tauri CLI 실행 |
| rust-analyzer | 최신 버전 | IDE Rust 언어 서버 |
| Tauri CLI | v2 계열 | `cargo install tauri-cli --locked` |
| Node.js | 20 LTS 이상 | Vite/Vue 개발 서버 실행 |
| pnpm | 최신 stable | 프로젝트 패키지 매니저 권장 |
| Vite | Vue 3 + TypeScript 템플릿 | `vue-ts` 템플릿 사용 |

### 3.2 OS별 필수 도구

| OS | 필수 도구 | 목적 |
|---|---|---|
| macOS | Xcode Command Line Tools | Rust native dependency 빌드, macOS 앱 패키징 |
| macOS | Apple Developer 계정 및 인증서 | 배포 서명, notarization 준비 |
| Windows | Visual Studio C++ Build Tools | MSVC toolchain 기반 Rust/Tauri 빌드 |
| Windows | Microsoft Edge WebView2 Runtime | Tauri WebView 런타임 |
| Windows | Windows SDK | Windows 앱 빌드 및 인스톨러 생성 |

## 4. macOS 개발 환경 설정

### 4.1 Xcode Command Line Tools 설치

```bash

```

설치 확인:

```bash

```

### 4.2 Rust 및 Cargo 설치

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
rustup update stable
rustup default stable
```

설치 확인:

```bash
rustc --version
cargo --version
rustup --version
```

### 4.3 rust-analyzer 설치

IDE 확장 기능으로 설치하는 방식을 기본으로 한다.

| IDE | 설치 방식 |
|---|---|
| Visual Studio Code | `rust-analyzer` 확장 설치 |
| JetBrains IDE | Rust 플러그인 또는 rust-analyzer 지원 기능 사용 |

CLI 설치가 필요한 경우:

```bash

```

### 4.4 Tauri CLI v2 설치

```bash
cargo install tauri-cli --locked
```

설치 확인:

```bash

```

### 4.5 Node.js 20 LTS 이상 및 pnpm 설치

```bash

```

```bash

설치 후 Developer PowerShell 또는 일반 PowerShell에서 확인:

```

### 5.2 WebView2 런타임 확인

Windows 10/11 최신 환경에는 일반적으로 Microsoft Edge WebView2 Runtime이 포함되어 있다. 누락된 경우 Microsoft 공식 WebView2 Runtime Evergreen Standalone Installer를 설치한다.

PowerShell 확인 예시:

```powershell
Get-ItemProperty "HKLM:\SOFTWARE\WOW6432Node\Microsoft\EdgeUpdate\Clients\*" |
  Where-Object { $_.name -like "*WebView2*" } |
  Select-Object name, pv
```

## 6. 프로젝트 초기화 명령어

신규 프로젝트 생성 시 Tauri 공식 생성기를 사용한다.

```bash
npm create tauri-app@latest -- --template vue-ts
```

생성 후 의존성 설치:

```bash

```

프로젝트가 npm 또는 yarn으로 초기화된 경우에도 운영 표준은 pnpm으로 통일한다. 패키지 매니저 혼용을 방지하기 위해 lockfile은 하나만 유지한다.

## 7. 빌드 및 실행 명령어

### 7.1 개발 실행

```bash
cargo tauri dev
```

### 7.2 운영 빌드

```bash
cargo tauri build
```

### 7.3 Windows 인스톨러 빌드

```bash
cargo tauri build --target x86_64-pc-windows-msvc
```

Windows 타깃 빌드는 Windows 빌드 머신에서 수행하는 것을 표준으로 한다. macOS 또는 Linux에서 Windows 빌드를 교차 컴파일하는 방식은 Tauri 공식 지원 범위가 아니며, 서명, WebView2, NSIS/MSI 인스톨러 도구 차이로 인해 운영 표준에서 제외한다.

Lugh 프로젝트의 Windows 배포 산출물은 GitHub Actions `windows-latest` runner에서 생성한다.

```bash
pnpm tauri build --bundles nsis,msi
```

| 산출물 | 경로 |
|---|---|
| NSIS 설치 파일 | `system/lugh/src-tauri/target/release/bundle/nsis/*.exe` |
| MSI 설치 파일 | `system/lugh/src-tauri/target/release/bundle/msi/*.msi` |

Windows 릴리즈 검증은 GitHub Actions artifact로 내려받은 `.exe`, `.msi`를 Windows 10/11 환경에서 설치 및 실행 확인하는 방식으로 수행한다.

### 7.4 macOS DMG 빌드

```bash
cargo tauri build
```

macOS 배포용 DMG는 macOS 빌드 머신에서 생성한다. 배포용 산출물은 Apple Developer 인증서 기반 코드 서명과 notarization 절차를 거친다.

## 8. CI/CD 환경

GitHub Actions를 기준 CI/CD 환경으로 사용한다. OS별 네이티브 패키징 요구사항 때문에 Windows 산출물은 Windows runner, macOS 산출물은 macOS runner에서 빌드한다.

### 8.1 Windows 빌드 머신 설정

| 항목 | 기준 |
|---|---|
| Runner | `windows-latest` 또는 프로젝트에서 고정한 Windows runner |
| Rust target | Windows runner 기본 MSVC target |
| Node.js | 22 |
| 패키지 매니저 | pnpm 10.7.1 |
| 필수 런타임 | WebView2 Runtime |
| Tauri build | `tauri-apps/tauri-action@v1` |
| 산출물 | NSIS `.exe`, MSI `.msi` |
| Workflow 초안 | `.github/workflows/build-windows.yml` |

빌드 단계 예시:

```yaml
name: Build Windows

on:
  workflow_dispatch:
  push:
    tags:
      - "lugh-v*"

permissions:
  contents: read

jobs:
  build-windows:
    name: Build Windows NSIS/MSI
    runs-on: windows-latest

    defaults:
      run:
        working-directory: system/lugh

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 10.7.1

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
          cache-dependency-path: system/lugh/pnpm-lock.yaml

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable

      - name: Install frontend dependencies
        run: pnpm install --frozen-lockfile

      - name: Build Tauri Windows bundles
        uses: tauri-apps/tauri-action@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          projectPath: system/lugh
          tauriScript: pnpm tauri
          args: --bundles nsis,msi

      - name: Upload NSIS installer
        uses: actions/upload-artifact@v4
        with:
          name: lugh-windows-nsis
          path: system/lugh/src-tauri/target/release/bundle/nsis/*.exe
          if-no-files-found: error

      - name: Upload MSI installer
        uses: actions/upload-artifact@v4
        with:
          name: lugh-windows-msi
          path: system/lugh/src-tauri/target/release/bundle/msi/*.msi
          if-no-files-found: error
```

초안 workflow는 수동 실행(`workflow_dispatch`)과 `lugh-v*` 태그 push를 지원한다. 릴리즈 자동 업로드가 필요하면 `permissions.contents`를 `write`로 조정하고 `tauri-action`의 `tagName`, `releaseName`, `releaseDraft` 입력을 추가한다.

### 8.2 macOS 빌드 머신 설정

| 항목 | 기준 |
|---|---|
| Runner | `macos-latest` 또는 프로젝트에서 고정한 macOS runner |
| Rust target | Apple Silicon 또는 Intel 배포 정책에 따라 결정 |
| Node.js | 20 LTS 이상 |
| 패키지 매니저 | pnpm |
| 필수 도구 | Xcode Command Line Tools |
| 산출물 | `.app`, `.dmg` |

빌드 단계 예시:

```yaml
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 20
- uses: pnpm/action-setup@v4
  with:
    version: latest
- uses: dtolnay/rust-toolchain@stable
- run: pnpm install --frozen-lockfile
- run: cargo tauri build
```

### 8.3 서명 및 notarization 준비 사항

| 대상 | 준비 항목 |
|---|---|
| Windows | 코드 서명 인증서, 인증서 비밀번호, 타임스탬프 서버 URL |
| macOS | Apple Developer Team ID, Developer ID Application 인증서, App Store Connect API Key, Issuer ID, Key ID |
| 공통 | 서명 비밀값을 GitHub Actions Secrets에 저장 |

비밀값은 git에 커밋하지 않는다. `.env`, 인증서 원본, private key, API key는 GitHub Actions Secrets 또는 배포 환경 secret manager를 통해 주입한다.

권장 GitHub Actions Secrets:

| Secret | 용도 |
|---|---|
| `WINDOWS_CERTIFICATE_BASE64` | Windows 코드 서명 인증서 |
| `WINDOWS_CERTIFICATE_PASSWORD` | Windows 인증서 비밀번호 |
| `APPLE_CERTIFICATE_BASE64` | macOS Developer ID 인증서 |
| `APPLE_CERTIFICATE_PASSWORD` | macOS 인증서 비밀번호 |
| `APPLE_TEAM_ID` | Apple Developer Team ID |
| `APPLE_API_KEY` | App Store Connect API Key |
| `APPLE_API_KEY_ID` | App Store Connect Key ID |
| `APPLE_API_ISSUER` | App Store Connect Issuer ID |

## 9. 표준 폴더 구조

Tauri + Vue 3 + TypeScript 프로젝트의 표준 구조는 다음과 같다.

```text
<project-root>/
├── package.json
├── pnpm-lock.yaml
├── index.html
├── vite.config.ts
├── tsconfig.json
├── src/
│   ├── main.ts
│   ├── App.vue
│   ├── assets/
│   ├── components/
│   ├── composables/
│   ├── router/
│   └── stores/
└── src-tauri/
    ├── Cargo.toml
    ├── build.rs
    ├── tauri.conf.json
    ├── capabilities/
    ├── icons/
    └── src/
        ├── main.rs
        └── lib.rs
```

| 경로 | 역할 |
|---|---|
| `src-tauri/` | Rust 백엔드, Tauri 설정, 권한, 네이티브 빌드 설정 |
| `src-tauri/src/` | Rust 애플리케이션 엔트리포인트와 커맨드 구현 |
| `src-tauri/Cargo.toml` | Rust crate 의존성 및 빌드 메타데이터 |
| `src-tauri/tauri.conf.json` | Tauri 앱 설정, 번들링, 보안 정책 |
| `src/` | Vue 3 프론트엔드 소스 |
| `src/components/` | Vue 컴포넌트 |
| `src/composables/` | Vue Composition API 공통 로직 |
| `src/assets/` | 프론트엔드 정적 자산 |
| `vite.config.ts` | Vite 빌드 및 개발 서버 설정 |

## 10. 운영 원칙

1. 개발자 로컬 환경은 Rust stable, Node.js 20 LTS 이상, pnpm 기준으로 통일한다.
2. OS별 패키징은 해당 OS runner에서 수행한다.
3. 실제 비밀값은 저장소에 커밋하지 않고 GitHub Actions Secrets 또는 환경변수로 주입한다.
4. `.env.example`에는 설명과 가짜 예시값만 기록한다.
5. 빌드 산출물은 CI/CD 아티팩트로 보관하고, 배포본은 서명 및 검증 절차를 거친다.
