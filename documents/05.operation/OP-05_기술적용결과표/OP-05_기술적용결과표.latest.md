## 1. 문서 개요

### 1.1 목적

본 문서는 `lugh` 프로젝트에 적용된 주요 기술 스택의 선정 사유, 적용 위치, 검증 결과, 리스크와 제약을 정리한다. 기준 아키텍처는 DS-20 아키텍처설계서의 Tauri v2 + Rust Backend + Vue 3 Frontend 구조와 정합을 유지한다.

### 1.2 대상 범위

| 항목 | 내용 |
|------|------|
| 대상 시스템 | `system/lugh` |
| 제품 성격 | AgiTeamBuilder 데스크톱 GUI |
| 주요 구조 | Tauri v2 런타임, Rust backend, Vue 3 + TypeScript frontend, Pinia state |
| 주요 연동 | Claude CLI/API, OpenAI, Gemini, Redmine REST API, OS Keychain/Credential Vault |
| 기준 문서 | DS-20 아키텍처설계서, DS-40 API명세서, DS-60 연동규격서 |

## 3. 기술 적용 결과표

### 3.1 Tauri v2 Desktop Runtime

| 구분 | 내용 |
|------|------|
| 선정 사유 | Rust backend와 Web frontend를 결합해 macOS/Windows 데스크톱 앱을 구성할 수 있고, 파일 시스템·OS credential·외부 브라우저/WebView 등 OS 연동을 backend command로 제한할 수 있다. |
| 적용 위치 | `system/lugh/src-tauri/src/lib.rs`, `system/lugh/src-tauri/tauri.conf.json`, `system/lugh/src-tauri/capabilities/default.json` |
| 적용 내용 | Tauri Builder에서 dialog/opener plugin, AppState, BrowserEmbedState, invoke handler를 등록한다. Vue frontend는 Tauri invoke/event API로 Rust command와 통신한다. |
| 검증 결과 | `cargo test` 통과로 Tauri command 포함 Rust crate 컴파일 확인. `pnpm run build` 통과로 Tauri가 로드할 frontend bundle 생성 확인. |
| 리스크·제약 | OS별 WebView 구현 차이가 있다. packaging, code signing, notarization, Windows installer 검증은 별도 배포 단계에서 확인해야 한다. |

### 3.2 Tauri WebviewWindow 임베디드 브라우저

| 구분 | 내용 |
|------|------|
| 선정 사유 | 앱 내부 사이드바/패널 영역에 외부 URL을 표시하면서도 main Vue 화면과 독립된 WebView 세션을 운용하기 위해 채택했다. |
| 적용 위치 | `src-tauri/src/commands/browser.rs`, `src-tauri/src/lib.rs`, `src/components/BrowserPanel.vue`, `src/stores/browser.ts` |
| 적용 내용 | `embedded-browser` WebviewWindow를 floating borderless 창으로 생성한다. `browser_open`, `browser_navigate`, `browser_back`, `browser_forward`, `browser_close`, `browser_resize` command를 제공하고, WebView navigation 시 `browser:navigation` event를 main 창에 emit한다. |
| 검증 결과 | `cargo test`로 browser command 포함 crate 컴파일 확인. `pnpm run build`로 `BrowserPanel.vue`와 browser store 타입/빌드 정합 확인. |
| 리스크·제약 | main window 기준 논리 좌표를 screen 좌표로 변환해 시각적 임베드를 구현하므로 OS별 창 위치, focus, minimize, always-on-top 동작 차이가 발생할 수 있다. 실제 드래그/리사이즈/포커스 회귀 테스트가 필요하다. |

### 3.4 reqwest HTTP Client

| 구분 | 내용 |
|------|------|
| 선정 사유 | Rust backend에서 AI provider와 Redmine REST API를 직접 호출하고, JSON 요청과 streaming 응답을 처리하기 위해 채택했다. |

### 3.5 keyring / OS Keychain Credential Vault

| 구분 | 내용 |
|------|------|
| 선정 사유 | API key와 Redmine key를 renderer나 파일에 평문 저장하지 않고 OS 보안 저장소에 위임하기 위해 채택했다. |
| 적용 위치 | `src-tauri/Cargo.toml`, `src-tauri/src/credential.rs`, `commands/credential.rs`, `commands/redmine.rs`, `stores/settings.ts` |
| 적용 내용 | `CredentialStoreService`가 provider/account 기준 save/get/delete/exists/masked 조회를 제공한다. Redmine은 `api_key_${role}` 우선, `api_key` fallback 규칙을 적용한다. |
| 검증 결과 | `cargo test`에서 masked credential, missing credential, save/get/delete 테스트 통과. OS keyring 접근이 불가능한 CI성 환경은 skip 처리 로직을 포함한다. |
| 리스크·제약 | macOS Keychain, Windows Credential Manager, CI/headless 환경별 동작 차이가 있다. 사용자 권한 프롬프트와 keychain 잠김 상태에 대한 운영 가이드가 필요하다. |

## 6. 결론

`lugh`에 적용된 핵심 기술 스택은 현재 구현에 반영되어 있으며, frontend build와 Rust test 기준으로 적용 결과는 **CONDITIONAL PASS**로 판정한다.
