### 3.4 산출물 기반 RACI 매트릭스

**원칙**: 모든 업무는 산출물 1개로 바인딩된다. "일을 했다" = "산출물을 작성했다". 아웃풋이 없으면 일하지 않은 것.

본 §3.4는 **`00.standard/RACI매트릭스.latest.md` 의 발췌 뷰**다. 전체 RACI(38건 매트릭스 산출물 + 12건 메타 레이어, R·A·C 분배·재료 열·정보공학 단계 등)는 본 매트릭스를 참조한다.

> **R** Responsible (수행·작성) / **A** Accountable (승인) / **C** Consulted (자문)
> ※ **I(Informed) 사용 안 함** (정보 흐름은 산출물 코드와 재료 열로 관리)
> ※ **하나의 산출물에는 하나의 R만 지정**

### 4.2 파일 명명 규칙

```
{ID}_{산출물명}.latest.{ext}

예:
  IN-10_수행계획서.latest.md          ← 현행본 (.latest 고정)
  AN-20_요구사항정의서.latest.md       ← 버전은 frontmatter
```

- 파일명에 날짜·버전 금지 — 현행본은 항상 `.latest` 고정 (링크 안 깨짐)
- 버전·날짜는 frontmatter(`version`·`last_updated`). 백업은 `{ID}_{산출물명}.{YYYYMMDDHHMMSS}.{ext}`
- 상세 규칙: `00.standard/RACI매트릭스.latest.md §3.2~§3.4`

## 6. 품질관리

### 6.1 품질보증 활동

역할별 테스트 책임:

| 주체 | 책임 |
|------|------|
| 개발자 (BE/FE) | 변경 부분 단위 테스트만 + 빌드·구동 확인 |
| QA | 전체 기능 회귀 테스트 (pytest + Playwright) + 산출물 정합성 감리 |
| PM | QA 검증 지시·결과 수령·결함 분배 |

검증 시점: ① 코드 수정 후 ② 릴리즈 전 ③ 설계 변경 후 ④ 환경 변경 후.

### 6.2 테스트 데이터 명세 체계

모든 QA 검증은 **시나리오 문서**를 사전 작성 후 수행.

**4대 운영 원칙**:
1. **재현성**: 사전 조건을 다른 사람이 문서만 보고 복원 가능
2. **감사성**: 문서만으로 PASS 근거 검증 가능
3. **오염 방지**: 원복 SQL로 명시해 다음 테스트 오염 차단
4. **역행 금지**: 기대값을 실행 **전에** 문서화 (실행 결과를 기대값으로 쓰지 않음)

시나리오 문서 5섹션: Given / When / Then / Actual / 판정+원복.

상세 템플릿: `03.management/02.상시관리/MG-40_품질관리대장/` 또는 `05.QA/테스트데이터/_템플릿.md`.

### 6.3 산출물 정합성 감리

코드 변경 시 다음 3종 설계 문서와의 정합성을 QA가 감리:

1. **API 명세서 (DS-40)** — 엔드포인트·요청/응답 스키마·상태코드
2. **DB 설계서 (DS-30)** — 테이블·컬럼·관계·ERD·제약조건·NULL (DS-30은 옛 ERD + 테이블명세서 통합)
3. **연동규격서 (DS-60)** — 외부 시스템 연계 인터페이스

불일치 발견 시 **품질관리대장(MG-40)** 에 등록 + Architect 수정.

### 6.4 품질관리 목표 및 검증 방안

| 항목 | 목표 |
|------|------|
| 테스트 커버리지 (백엔드 API) | 80% 이상 |
| 빌드 검증 | 프론트·백엔드 모두 성공 필수 |
| 문법 오류 | 컴파일 에러 0건 |
| 설계-코드 정합성 | 품질관리대장 열림 건 0 (릴리즈 기준) |

## 8. 보안관리

### 8.1 보안관리 적용 범위

- API 키·비밀번호·JWT secret 등 민감 값
- DB 연결 정보·SSH 접근 키
- 고객 개인정보·민감 비즈니스 데이터

### 8.2 책임과 역할

| 영역 | 담당 |
|------|:---:|
| 비밀값 관리·환경변수 | DevOps |
| 코드 내 암호화 구현 | DeveloperBE |
| 접근 통제·권한 검증 | DeveloperBE + DevOps |
| 보안관리대장 (MG-50) | DevOps |

### 8.3 비밀값 관리 원칙

- **실제 비밀값을 git에 커밋 금지** (`.env`는 `.gitignore` 필수)
- **`.env.example`에는 설명 + 가짜 예시값만**
- 프로덕션 비밀값은 secret manager 또는 환경변수 주입
- 페르소나 인스턴스의 API 키는 `brain/` 폴더 단위로 git 비공유

### 8.4 API 키·토큰 관리

| 유형 | 보관 위치 | 회전 주기 |
|------|---------|:---:|
| 레드마인 API 키 | `brain/<역할>/persona.md` (인스턴스, git 비공유) | 필요 시 |
| JWT secret | `.env` (서버 환경변수) | 3개월 |
| DB 비밀번호 | `.env` | 3개월 |

### 8.5 접근 제어

- **SSH 접근**: DevOps만 운영서버 SSH 키 보유, 기타 역할은 PM 경유
- **DB 직접 접근**: 원칙상 금지. API 경유. 예외 시 DevOps가 일시 접근권 부여
- **감사 로그**: 운영서버 접근·DB 변경 감사 로그 수집

### 8.6 감사 로그

- 로그인·로그아웃·실패 시도
- 민감 데이터 접근 (개인정보 조회·수정)
- 관리자 권한 승격
- 배포·환경변수 변경

## 9. 페르소나·원형 관리 (AI 에이전트 팀 고유)

### 9.1 7역할 원형 참조

모든 페르소나 인스턴스는 **AgiTeamBuilder 원형**을 상속한다.

```
원형 (불변 DNA):    AgiTeamBuilder/brain/<역할>/persona.md
인스턴스 (프로젝트): <본 프로젝트>/brain/<역할>/persona.md
```

각 역할의 원형 내용은 **AgiTeamBuilder 저장소**에서 관리되며, 본 프로젝트는 이를 상속받아 고유 상태만 인스턴스에 추가한다.

### 9.2 인스턴스(persona.md) 관리 규칙

인스턴스 persona.md에 담을 내용:
- 프로젝트 개요·현재 Phase·미완료·알려진 이슈
- 팀원 surface 매핑·별칭
- 레드마인 API 키·user_id (프로젝트별)
- 운영 환경 정보 (서버 IP·도메인·내부 엔드포인트)
- 핵심 의사결정 이력
- 프로젝트 도메인 지식

### 9.3 자가 업데이트 사이클

```
1. 세션 중 교훈 발생
2. 담당 에이전트가 인스턴스 persona.md에 기록
3. Phase 종료 시 PM이 회고 문서에 정리
4. 여러 프로젝트에서 반복 검증된 교훈은 AgiTeamBuilder 원형으로 증류 (PM 권한)
5. 차세대 프로젝트 인스턴스가 갱신된 원형을 상속
```

### 9.4 증류 판정 기준

교훈이 AgiTeamBuilder 원형으로 승격되려면:

1. **재현성**: 최소 2개 이상 프로젝트에서 동일 문제 발생
2. **일반성**: 특정 도메인·기술스택에 종속되지 않음
3. **실효성**: 원칙 적용으로 실제 개선 확인됨
