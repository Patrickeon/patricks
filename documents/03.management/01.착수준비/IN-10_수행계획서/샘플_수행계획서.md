## 1. 프로젝트 개요

### 1.1 목적
1인 기업 및 소규모 조직의 경영지원(HR, 재무, 영업지원) 업무를 자동화하고 효율화하기 위한 통합 솔루션을 개발한다.

### 1.2 범위

| 모듈 | 형태 | 대상 사용자 | 주요 기능 |
|------|------|-----------|----------|
| 모듈 A | 웹 애플리케이션 | 전 직원 | 휴가 신청/조회/승인, 근태 관리 |
| 모듈 B | 웹 애플리케이션 | 관리자/대표 | 계약관리, 매입/매출 관리, 회사 지출 관리, CRM, 대시보드 |

### 1.3 Phase 구분

| Phase | 내용 | 우선순위 |
|-------|------|---------|
| Phase 1 | 웹 기반 휴가 관리 (직원 포털 + 관리자 백오피스) | ★★★ |
| Phase 2 | 계약서 템플릿 및 자동완성, 고객사 CRM | ★★☆ |
| Phase 3 | 매입/매출 세금계산서 관리, 회사 지출 관리, 월간 현금흐름 예측 | ★★☆ |
| Phase 4 | 통합 대시보드 및 알림 시스템 | ★☆☆ |

### 1.4 Phase 3 상세 요구사항

#### 1.4.1 매입 관리 (지급 스케줄)
- 거래처별 매입 세금계산서 수령 시 등록 (금액, 날짜)
- 거래처별 지급 예정일/금액 스케줄 관리
- 거래처는 Phase 2 고객사(clients) 테이블을 재활용 (매입처/매출처 구분 없이 동일 거래처가 둘 다 가능)
- 월별 매입 합계 조회

#### 1.4.2 매출 관리 (발행 스케줄)
- 거래처별 세금계산서 발행 스케줄 등록 (금액, 날짜)
- 거래처별 입금 예상일/금액 관리
- 월별 매출 합계 조회

#### 1.4.3 매입/매출 합산
- 월별 매입 합계 + 매출 합계 대시보드
- 손익 요약 (매출 - 매입)

#### 1.4.4 회사 지출 관리
- 세금계산서 외 일반 경비 등록 (임대료, 급여, 접대비, 소모품 등)
- 카테고리 분류: 고정비 / 변동비
- 월별 지출 합계 조회
- 매입 + 매출 + 지출 전체 합산 현금흐름 요약

#### 1.4.5 다음달 현금흐름 예측
- 지난달 매입/매출/지출 데이터를 단순 반복하여 이번달 예상 현금흐름 산출
- 예상 수입 (매출 입금 예상), 예상 지출 (매입 지급 + 회사 지출), 예상 잔액

#### 1.4.6 부가세 신고 보조
- 분기별 매입/매출 세금계산서 합계 리포트
- 부가세 예상액 계산 (매출세액 - 매입세액)

### 1.5 Phase 4 상세 요구사항

#### 1.5.1 직원 대시보드
- 내 휴가 현황 (잔여/사용/진행중 신청)
- 이번주 팀 휴가자 리스트
- 내 최근 알림 5건
- 공지사항/메모 위젯
- 빠른 액션 버튼 (휴가 신청 바로가기)

#### 1.5.2 관리자 대시보드
- 승인 대기 건수 + 바로가기
- 오늘/이번주 휴가자 리스트
- 계약 현황 (서명대기, 이번달 만료 예정, 최근 체결)
- 이번달 현금흐름 요약 (매출/매입/지출/순이익 카드 4개)
- 입금/지급 예정 리스트 (이번주 기준)
- 월별 추이 차트 (최근 6개월 매출/매입)

#### 1.5.3 알림 센터
- 인앱 알림 이력 조회 페이지
- 알림 읽음/안읽음 관리
- 알림 유형별 필터

#### 1.5.4 이메일 알림
- 이벤트 기반 알림 6종:
  - 휴가 신청됨 → 관리자
  - 휴가 승인/반려 → 신청자
  - 계약서 상태 변경 → 관리자
  - 계약 만료 임박 (7일 전) → 관리자
  - 입금예정일 도래 → 관리자
  - 지급예정일 도래 → 관리자
- 일일 요약 이메일: 매일 아침 어제 일어난 일 + 오늘 할 일 요약
- 알림 설정 페이지에서 이메일 수신 ON/OFF 가능

#### 1.5.5 알림 설정
- 사용자별 이메일 알림 수신 ON/OFF 설정
- 알림 유형별 개별 설정 가능

## 6. 작업 수행 프로세스

> 상세 태스크 분해(Step 1~6), 산출물 의존관계, 버그 수정/설계 변경 프로세스는 `documents/관리/WBS.md`로 이동하였다.

### 개발 방법론
Phase별 미니 워터폴: **분석 → 설계 → 환경 → 개발 → 검증 → 배포**

> **핵심 원칙: PM은 에이전트에게 설계 문서 경로와 해당 섹션을 명시하여 지시한다.**
> 예: "A-04 API명세서 3.2절의 POST /api/leaves를 구현해. 스키마는 A-03 테이블정의서 2.1절 참고."

## 7. 품질 관리

### 7.1 코드 품질 기준

| 항목 | 기준 |
|------|------|
| 코드 리뷰 | 모든 코드는 QA 리뷰 후 반영 |
| 테스트 커버리지 | 백엔드 API: 80% 이상 목표 |
| 빌드 검증 | 프론트엔드 빌드 성공 필수 |
| 문법 오류 | 백엔드 Python 컴파일 에러 0건 |

### 7.2 산출물 품질 기준

| 항목 | 기준 |
|------|------|
| 설계 문서 | 코드 구현 전 반드시 작성 및 승인 |
| 문서-코드 일치 | 코드 변경 시 관련 문서 동시 갱신 |
| QA 검증 | 설계 문서 기준으로 검증, PASS/FAIL 명시 |
