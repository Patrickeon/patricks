# 품질관리대장 (Quality Management Register)

> 작성일: 2026-04-05
> 작성자: PM (박피엠)
> 감리 참여: QA (홍감리), Architect (김아키), PM 직접 탐색
> 대상: Phase 1~4 전체 산출물 + 코드

---

## 1. 개요

본 대장은 코드와 산출물 간 정합성 감리 결과를 기록하고, 이슈별 조치 상태를 추적한다.

### 감리 방법
- PM: 산출물 리스트 vs 실제 파일 대조, 화면기획서 URL vs 프론트엔드 라우트 대조
- QA: API 명세서 vs 실제 엔드포인트, 테이블 정의서 vs 모델, 산출물 경로 검증
- Architect: ERD vs ORM, 설계 문서 내부 정합성, 권한 매트릭스 vs 실제 구현

---

## 2. 작성 방법

### 심각도 기준
- 🔴 **상**: 설계-코드 불일치, 산출물 누락, 보안 관련
- 🟡 **중**: 문서 오류, 수치 미갱신, 설명 불일치
- 🟢 **하**: 경미한 표기 오류, 개선 권장

### 상태 값
- **열림**: 발견됨, 미조치
- **진행중**: 담당자가 조치 중
- **완료**: 조치 완료 및 검증됨

### 컬럼 정의

| 컬럼 | 설명 | 작성 예시 |
|------|------|----------|
| QI ID | 순번 ID (QI-001부터 순차) | QI-016 |
| 발견일 | 이슈 발견 날짜 (MM-DD) | 04-06 |
| 발견자 | 발견한 역할명 | QA, PM, Architect, 유저 |
| 대상 | 불일치가 발생한 문서 또는 코드 | API명세서 (계약서), ERD |
| 유형 | 이슈 분류 | 설계-코드 불일치, 문서 오류, 누락, 코드 결함 |
| 내용 | 불일치 내용 요약 (문서 기술 vs 실제 구현) | 간결하게 핵심만 기술 |
| 심각도 | 상/중/하 | 🔴 상 |
| 담당자 | 조치 책임자 | Architect, DeveloperBE 등 |
| 조치 내용 | 실제 수행한 조치 (완료 시 기입) | 미조치 시 `-` |
| 상태 | 열림 / 진행중 / 완료 | 열림 |

### 작성 규칙
1. 새 이슈는 기존 테이블 하단에 추가한다 (별도 섹션을 만들지 않는다)
2. QI ID는 기존 마지막 번호 + 1로 순차 부여한다
3. 내용은 한 셀에 간결하게 요약한다 (장문 금지)
4. 조치 완료 시 조치 내용 컬럼을 채우고 상태를 완료로 변경한다
5. 하단 요약(심각도별, 진행 현황)도 함께 갱신한다
