# lugh E2E 테스트 자동화 도입 조사 보고서

## 1. 배경 및 목적

- 현재 lugh(Tauri v2 + Rust + Vue3)는 브라우저 패널·첨부·대화 초기화 등 다수 기능을 **유저 육안 확인**으로만 검증 중 (#20~#24 대기 이력, #17도 3차 수정 끝에 육안 확인으로 종결).
- 육안 의존을 줄이고 회귀를 자동 검증하기 위한 E2E 테스트 자동화 도입 실현가능성을 조사한다.
- **본 문서는 조사·권고만 수행하며, 코드/설정은 변경하지 않았다.**
- 조사는 WebSearch/WebFetch로 2026-07 시점 최신 정보를 확인했다. localhost/네트워크 제약은 없었음(모든 조회 정상 응답). 사용량 한도 도달 사실 없음.

## 2. 조사 결과 요약 (한눈에)

| 항목 | 결론 |
|---|---|
| Linux(CI) E2E | ✅ 성숙·저비용·즉시 도입 가능 (webkit2gtk-driver + xvfb) |
| Windows(CI) E2E | ✅ 가능, 자동 드라이버 관리 (msedgedriver), 기존 build-windows.yml 인프라 재사용 가능 |
| macOS E2E | ⚠️ 가능하나 **핵심 축(@wdio/tauri-service, tauri-plugin-wdio)이 `1.0.0-next.0` 프리릴리즈** — 안정성 리스크 있음, 앱에 플러그인 추가 필요 |
| Playwright | ⚠️ Tauri 표준 CDP 미지원(WebView2=Windows만 네이티브) — 서드파티 브릿지(tauri-plugin-playwright) 존재하나 0.x 초기 단계 |
| 브라우저 패널(#17) 테스트 | ⚠️ iframe이 아닌 **별도 네이티브 floating window** — 표준 단일창 예제로 커버 불가, 멀티윈도우 스파이크 필요 |
| FE 준비 상태 | ⚠️ `data-testid` 등 안정적 셀렉터 전무(0건) — 도입 전 FE 협업 필요 |
| 권고 | **Linux 헤드리스 스모크부터 단계적 도입**, macOS/브라우저패널은 후순위 |

## 3. 조사 항목별 상세

### 3.1 Tauri v2 공식/권장 E2E 방법

- 공식 권장 도구는 **WebdriverIO + `@wdio/tauri-service`**. Windows/Linux/macOS 3개 플랫폼을 모두 지원한다고 명시됨.
- Node.js를 쓰지 않거나 Selenium을 선호하면 **`tauri-driver`를 직접 구동**하는 방식도 공식 문서에 있음. 단, 이 경로는 **"Windows와 Linux만 지원 — macOS는 WKWebView용 WebDriver 드라이버 자체가 없다"**는 제약이 명시돼 있음.
- macOS는 `@wdio/tauri-service`의 **"embedded" 드라이버 provider**(앱 안에서 자체 WebDriver 서버를 구동)로만 지원되며, 이를 위해 앱에 `tauri-plugin-wdio-webdriver`(Rust 플러그인)를 추가해야 한다. 즉 macOS E2E는 **테스트 도구가 아니라 앱 코드에 플러그인을 심어야 하는 구조** — 순수 "테스트 인프라 추가"보다 침습적이다.
- **버전 성숙도 확인**: `@wdio/tauri-service` npm 최신판은 `1.0.0-next.0`(pre-release), Cargo 쪽 `tauri-plugin-wdio`도 `1.0.0-next.0` — **GA(정식) 릴리즈 이전 단계**. breaking change 가능성을 감안해야 함.
- 플랫폼별 드라이버 메커니즘 요약:

| 플랫폼 | 드라이버 | 자동 관리 | 비고 |
|---|---|---|---|
| Windows | Microsoft Edge WebDriver(msedgedriver) ↔ WebView2 | 자동 다운로드/버전매칭 | 별도 앱 코드 변경 불필요 |
| Linux | WebKitWebDriver(webkit2gtk) | 수동 설치(apt 등) | Ubuntu/Fedora40+/Arch만 공식 지원, Alpine/CentOS·RHEL 미지원 |
| macOS | embedded 서버(앱 내장, `tauri-plugin-wdio-webdriver` 필요) | 자동 | **프리릴리즈**, 앱 코드에 플러그인 추가 필요 |

Sources: [WebDriver | Tauri](https://v2.tauri.app/develop/tests/webdriver/), [Tests | Tauri](https://v2.tauri.app/develop/tests/), [Platform Support | WebdriverIO](https://webdriver.io/docs/desktop-testing/tauri/platform-support/), [WebdriverIO | Tauri](https://v2.tauri.app/develop/tests/webdriver/example/webdriverio/), [Selenium | Tauri](https://v2.tauri.app/develop/tests/webdriver/example/selenium/), [@wdio/tauri-service - npm](https://www.npmjs.com/package/@wdio/tauri-service), [tauri-plugin-wdio - Libraries.io](https://libraries.io/cargo/tauri-plugin-wdio)

### 3.2 대안 프레임워크 — Playwright 및 기타

- **표준 Playwright는 Tauri와 구조적으로 맞지 않음**: Playwright는 Chrome DevTools Protocol(CDP)로 브라우저를 제어하는데, Tauri의 3개 플랫폼 웹뷰 중 **CDP를 네이티브 지원하는 것은 Windows의 WebView2뿐**. macOS(WKWebView)·Linux(WebKitGTK)는 CDP 자체가 없어 표준 Playwright 연결이 불가능.
- 서드파티 브릿지 **`tauri-plugin-playwright`**가 존재: 네이티브 웹뷰에 소켓 브릿지를 붙여 Playwright 호환 API(오토웨이팅, locator, 스크린샷 등)를 제공. 3가지 모드(headless Chromium+IPC mock / 소켓 브릿지 실웹뷰 / Windows 전용 직접 CDP) 지원. 다만 **Rust crate 버전이 0.1~0.3대**로 초기 단계이며, 표준 WebDriver 프로토콜이 아닌 **커스텀 소켓 아키텍처**라 생태계 지원(리포터, CI 통합 등)이 WebdriverIO 대비 얕음.
- 결론: **Playwright는 "친숙함" 대비 Tauri 적합성이 낮음** — 현재 시점(2026-07)에는 공식 경로인 WebdriverIO가 더 안전한 선택.

Sources: [tauri-plugin-playwright - crates.io](https://crates.io/crates/tauri-plugin-playwright), [tauri-playwright README](https://github.com/srsholmes/tauri-playwright/blob/main/README.md), [Playwright Testing: WebKit Engine Mismatch](https://takazudomodular.com/pj/zudo-tauri/docs/frontend/playwright-engine-pitfall/)

### 3.3 CI(GitHub Actions) 실행 가능성

공식 Tauri CI 가이드 기준(예제는 `ubuntu-latest`+`windows-latest` 매트릭스, macOS는 가이드에 미포함):

| 플랫폼 | 헤드리스/디스플레이 요구 | 필요 패키지 | 러너 과금 배율 |
|---|---|---|---|
| Linux (ubuntu-latest) | **xvfb 필수** (가상 디스플레이로 `xvfb-run` 접두) | `libwebkit2gtk-4.1-dev`, `webkit2gtk-driver`, `xvfb` 등 (기존 build 의존성 + 추가 2종) | **1x** (가장 저렴) |
| Windows (windows-latest) | 불필요 (직접 실행 가능) | msedgedriver (cargo tool로 설치) | 2x |
| macOS (macos-latest) | 별도 가이드 없음(자체 검증 필요) | `tauri-plugin-wdio-webdriver` 앱 내장 | **10x** (가장 비쌈) |

- 2026-01 기준 분당 요금: Linux $0.006, Windows $0.010, macOS $0.062 (공개 레포는 무료). 사설 레포 기준 macOS는 Linux의 약 10배 비용.
- lugh는 이미 `build-windows.yml`(windows-latest)이 있어 **Windows E2E는 기존 인프라 확장으로 저비용 추가 가능**.
- Linux는 지금 프로젝트에 없는 새 러너지만 **가장 싸고 성숙도 높은 경로** — E2E 최초 도입 대상으로 적합.
- macOS CI E2E는 비용·프리릴리즈 리스크 둘 다 높아 **후순위** 권고(3.5절 참조).

Sources: [Continuous Integration | Tauri](https://v2.tauri.app/develop/tests/webdriver/ci/), [GitHub Actions Runner Pricing 2026](https://cicdpipelinecost.com/github-actions-pricing), [GitHub Docs - Actions runner pricing](https://docs.github.com/en/billing/reference/actions-runner-pricing)

### 3.4 lugh 적용 시 우선 커버 시나리오 후보

Rust 백엔드 IPC 커맨드(`#[tauri::command]`) 33개를 실제 코드에서 확인, 화면과 매핑:

| 우선순위 | 시나리오 | 관련 IPC 커맨드 | 화면 | 비고 |
|:---:|---|---|---|---|
| 1 (스모크) | 앱 부팅 → 헬스체크 통과 | `boot_team`/`boot_role`, `run_health_check` | BootView | 단일창, 가장 단순 — Phase 1 대상 |
| 1 (스모크) | 워크스페이스 열기·검증 | `open_workspace`, `load_workspace_config`, `validate_workspace` | LauncherView/WorkspaceView | 단일창 |
| 2 | 채팅 메시지 송수신 | `send_agent_message`, `list_agent_messages`, `get_agent_session` | WorkspaceView | 실제 LLM 응답 대신 **IPC mock 권장**(비용·지연·flaky 방지) |
| 2 | 대화 초기화(#22 관련 이력) | `clear_session_messages` | WorkspaceView | 상태 리셋 검증에 적합, 회귀 잘 잡힘 |
| 2 | 첨부(#21) | `prepare_chat_attachment` | WorkspaceView | 파일 업로드 다이얼로그 자동화 필요(OS 네이티브 다이얼로그는 WebDriver로 직접 제어 불가 — 플러그인 mock 또는 파일 경로 직접 주입 방식 검토 필요) |
| 3 (난이도 高) | 브라우저 열기/이동/뒤로가기(#17) | `browser_open`, `browser_navigate`, `browser_back/forward`, `browser_close` | BrowserPanel | **iframe이 아닌 별도 floating borderless 네이티브 창** — WebDriver 멀티윈도우(window handle switch) 스파이크 필요, 공식 예제 문서에 멀티윈도우 시나리오 없음(확인됨) |
| 4 | Redmine 연동 (조회/생성/수정) | `redmine_list_issues` 등 | RedmineView | 로컬 Docker Redmine 의존 — CI에서 Redmine 컨테이너 동반 기동 필요(docker-compose), 별도 인프라 작업 |
| 4 | 자격증명 저장/조회 | `save_credential`, `validate_credential`, `delete_credential` | ProjectSettingsView | **기보고 keyring mock-backend 이슈와 연결**: 현재 Cargo.toml에 OS 백엔드 feature 미지정이라 CI에서는 항상 mock으로 동작 → E2E는 "통과"해도 **실제 OS Keychain/Credential Manager 저장 여부는 검증되지 않음**을 인지 필요 |

권고 순서: **1(부팅/워크스페이스) → 2(채팅/초기화/첨부) → 4(자격증명, mock 한계 인지 하 저비용 커버) → 3(브라우저 패널, 스파이크 후 판단)**. Redmine 연동은 인프라 부담 대비 가치가 낮아 초기 스코프에서 제외 권고.

### 3.5 도입 비용/난이도 평가 + 단계적 도입안

**사전 준비 비용(공통, 반드시 선행)**:
- FE에 `data-testid` 등 안정 셀렉터 부재(0건 확인) → 셀렉터 전략 없이 시작하면 테스트가 CSS 클래스/텍스트에 의존해 매우 깨지기 쉬움. **BE/FE 협업으로 핵심 컴포넌트에 `data-testid` 추가가 선행 과제**.
- E2E 테스트 코드/픽스처 작성은 QA 영역(RACI), CI 파이프라인은 DevOps 영역 — 역할 분담 PM 확정 필요.

**단계적 도입안**:

| Phase | 범위 | 러너 | 목표 | 리스크/비고 |
|---|---|---|---|---|
| **0. 스파이크** | WebdriverIO 공식 예제 그대로(타이틀 텍스트 확인 1건) | ubuntu-latest + xvfb | 인프라(webkit2gtk-driver 설치, wdio 설정) 동작 확인만 | 실패해도 무방, 1~2일 규모 |
| **1. 최소 스모크 상시화** | 부팅 + 워크스페이스 열기 | ubuntu-latest | PR/main push마다 자동 실행(`e2e-linux.yml` 신설) | 가장 저비용(1x), 우선 도입 대상 |
| **2. 핵심 플로우 확대** | 채팅 송수신(mock)·대화 초기화·첨부 | ubuntu-latest | 회귀 커버리지 확대 | LLM 실호출 대신 IPC mock 설계 필요(BE 협업) |
| **3. Windows 확장** | Phase 1~2 시나리오 Windows 재실행 | windows-latest (기존 build-windows.yml 인프라 재사용) | 크로스플랫폼 회귀 확보 | msedgedriver 자동 관리라 난이도 낮음, 비용 2x |
| **4. 브라우저 패널(#17)** | 멀티윈도우 시나리오 | ubuntu-latest 우선 | 별도 스파이크로 멀티윈도우 핸들링 가능 여부 먼저 검증 | 공식 문서 커버 안 됨 — 실패 시 수동 육안 검증 유지 |
| **5. macOS 확장(신중)** | Phase 1 스모크만 우선 | macos-latest | `@wdio/tauri-service` GA 전환 후 재검토 권고 | 10x 비용 + 프리릴리즈 리스크, `tauri-plugin-wdio-webdriver` 앱 코드 추가 필요 — **당장 도입 비권고**, GA 및 안정화 확인 후 착수 |

**개략 난이도/비용 등급**:

| 축 | 난이도 | CI 비용 | 안정성(현시점) |
|---|:---:|:---:|:---:|
| Linux 헤드리스 | 낮음 | 매우 낮음 | 높음(성숙 도구) |
| Windows | 낮음~중간 | 중간 | 높음(자동 드라이버 관리) |
| macOS | 높음(앱 코드 변경 수반) | 매우 높음 | **낮음(프리릴리즈)** |
| 브라우저 패널(멀티윈도우) | 높음(레퍼런스 부재) | - | 미검증 |

## 4. 리스크 및 제약사항 종합

1. **`@wdio/tauri-service`/`tauri-plugin-wdio` 프리릴리즈(1.0.0-next.0)** — macOS 경로 전체가 여기 의존, breaking change 리스크.
2. **브라우저 패널이 iframe이 아닌 네이티브 floating window** — 표준 단일창 WebDriver 예제로 커버 안 됨, 별도 스파이크 필요.
3. **FE 셀렉터 부재(data-testid 0건)** — 선행 작업 없이는 테스트가 깨지기 쉬움.
4. **keyring mock-backend 기보고 이슈**와 연결 — CI E2E에서 자격증명 저장 시나리오가 "통과"해도 실제 OS 보안 저장소 검증은 아님(별개로 BE 조치 필요 사안, 레드마인 #22 보고서에서 이미 언급).
5. **Redmine 연동 시나리오는 CI에 Docker Redmine 동반 기동 필요** — 인프라 부담 대비 우선순위 낮음.
6. **LLM 실제 API 호출 시나리오**는 비용·지연·flaky 리스크 — IPC mocking 설계가 선행돼야 함(BE 협업 필요, 본 조사 범위 밖).
7. macOS는 러너 비용이 Linux의 약 10배 — 빈도 제한(예: 릴리즈 태그 시에만) 권장.

## 5. 권고 요약

1. **Phase 0~1(Linux 헤드리스 스모크)부터 즉시 착수 가능** — 비용·난이도·성숙도 모두 유리.
2. macOS E2E는 **`@wdio/tauri-service` GA 전환 전까지 보류** 권고. lugh 팀 로컬 개발 환경이 macOS인 점을 고려하면 가치는 크지만, 지금 도입하면 프리릴리즈 불안정성을 그대로 안게 됨.
3. Playwright 대신 **공식 경로(WebdriverIO)를 기본 전제**로 추진.
4. 브라우저 패널(#17류) E2E는 **별도 멀티윈도우 스파이크**로 실현 가능성만 먼저 확인 후 범위 포함 여부 결정.
5. 착수 전 **FE `data-testid` 부여 + BE IPC mock 설계**를 선행 과제로 PM이 BE/FE에 지시 필요 (DevOps 단독 착수 불가 — 코드 변경 수반).
6. 본 조사는 **권고까지만**이며, 실제 도입(워크플로우 신설, 패키지 설치, 플러그인 추가)은 **PM 승인 후** 별도 작업으로 진행 권장.
